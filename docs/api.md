# Workflow API

## Overview

The plasmoid does not perform any workflow actions itself.

Instead, it launches an external workflow implementation and monitors its status through a simple command-line API.

The workflow implementation may be written in any language and may perform any actions required before logout, reboot, or shutdown.

Examples:

* rsync
* cloud synchronisation
* backup verification
* container shutdown
* virtual machine shutdown
* custom user scripts

The plasmoid treats the workflow implementation as a black box.

---

## Commands

### Start workflow

```bash
workflow start shutdown
workflow start reboot
workflow start logout
```

Response:

```json
{
  "id": "7b0a7d8f",
  "on_error": "terminate"
}
```

### Query workflow status

```bash
workflow status 7b0a7d8f
```

### Cancel workflow

```bash
workflow cancel 7b0a7d8f
```

---

## Actions

Supported workflow actions:

```text
shutdown
reboot
logout
```

These correspond to the user's original selection in the plasmoid.

---

## States

Supported workflow states:

```text
running
success
error
cancelled
```

### running

The workflow is currently executing.

### success

The workflow completed successfully.

The plasmoid should continue with the requested power action.

### error

The workflow failed.

The plasmoid behaviour is determined by the `on_error` policy.

### cancelled

The workflow was cancelled before completion.

The plasmoid must not continue with the requested power action.

---

## Error Policies

Supported values:

```text
terminate
proceed
ask
```

### terminate

Errors are considered critical.

The requested power action must not proceed automatically.

The user must choose:

```text
Force Action
Cancel
```

### proceed

Errors are considered non-critical.

The plasmoid should display a warning and continue automatically after a countdown.

### ask

The workflow delegates the decision to the user.

The plasmoid should ask whether to continue or cancel.

---

## JSON Schema

### Required Fields

The following fields are mandatory in every valid status response:

```json
{
  "id": "7b0a7d8f",
  "state": "running",
  "on_error": "terminate"
}
```

All other fields are optional.

---

### id

Type:

```text
string
```

Example:

```json
{
  "id": "7b0a7d8f"
}
```

Unique workflow identifier.

The identifier is generated by the workflow implementation during the `start` command.

The same identifier must be used in all subsequent `status` and `cancel` operations.

---

### state

Type:

```text
string
```

Allowed values:

```text
running
success
error
cancelled
```

Represents the current workflow state.

---

### on_error

Type:

```text
string
```

Allowed values:

```text
terminate
proceed
ask
```

Defines how the plasmoid should behave if the workflow enters the `error` state.

The workflow should provide this value during `start`.

The value may also be repeated in status responses.

---

### workflow_name

Type:

```text
string
```

Example:

```json
{
  "workflow_name": "Shutdown Workflow"
}
```

Human-readable workflow name displayed by the plasmoid.

---

### step_name

Type:

```text
string
```

Example:

```json
{
  "step_name": "Waiting for the Process"
}
```

Human-readable description of the current workflow step.

---

### step_current

Type:

```text
integer
```

Example:

```json
{
  "step_current": 2
}
```

Current step number.

Step numbering starts at 1.

---

### step_total

Type:

```text
integer
```

Example:

```json
{
  "step_total": 5
}
```

Total number of workflow steps.

---

### step_percent

Type:

```text
integer
```

Range:

```text
0-100
```

Example:

```json
{
  "step_percent": 35
}
```

Progress of the current step.

---

### workflow_percent

Type:

```text
integer
```

Range:

```text
0-100
```

Example:

```json
{
  "workflow_percent": 63
}
```

Overall workflow progress.

If provided, this value takes precedence over any progress estimation performed by the plasmoid.

---

### started

Type:

```text
ISO 8601 timestamp
```

Example:

```json
{
  "started": "2026-06-07T03:12:44Z"
}
```

Workflow start time.

May be used by the plasmoid to display elapsed time and estimate completion time.

---

### message

Type:

```text
string
```

Example:

```json
{
  "message": "Synchronisation timed out"
}
```

Human-readable informational or error message.

Most commonly used together with the `error` state.

---

## Progress Calculation

If `workflow_percent` is present, the plasmoid must use it as the primary workflow progress indicator.

If `workflow_percent` is absent, the plasmoid may estimate overall progress using:

* `step_current`
* `step_total`
* `step_percent`

If both `workflow_percent` and `step_percent` are present:

* `workflow_percent` should be used for the main progress bar.
* `step_percent` may be displayed as current-step progress.

## Design Principles

* The plasmoid must not know what the workflow actually does.
* The workflow may be implemented in any language.
* All communication is performed through command-line calls and JSON responses.
* KDE, Plasma, DBus, sockets, and system services are not required.
* Only a minimal set of fields is mandatory.
* Progress reporting is optional.
* The workflow implementation remains fully independent from KDE and Plasma.
* The API should remain stable across future workflow implementations.

---

## Implementation Notes

The following recommendations are not part of the API specification but may simplify workflow implementation.

### State Storage

The workflow implementation is responsible for persisting its own state between invocations.

Typical locations include:

```text
/run/user/<uid>/plasma-power-workflow/
~/.cache/plasma-power-workflow/
```

The plasmoid does not access workflow state files directly.

All communication occurs through the command-line API.

### Polling Frequency

Workflow status should not be queried excessively.

A polling interval between 0.5 and 2 seconds is recommended.

The exact interval is implementation-dependent.

### Invalid JSON Responses

Workflow implementations may update their state asynchronously.

As a result, temporary invalid or incomplete JSON responses may occasionally occur.

Clients should treat JSON parsing failures as temporary conditions and retry later rather than immediately reporting an error to the user.

### Atomic State Updates

Workflow implementations are encouraged to update status files atomically.

A common approach is:

1. Write the new status to a temporary file.
2. Rename the temporary file over the previous status file.

This prevents clients from observing partially written JSON documents.

### Progress Reporting

Providing progress information is optional.

Implementations may report:

* only workflow state;
* step-based progress;
* overall workflow progress;
* both step-level and workflow-level progress.

The plasmoid should remain functional even when no progress information is available.

### Workflow Lifetime

Workflow implementations are not required to run as persistent services.

A common pattern is:

1. `workflow start` launches a background task.
2. The task stores its status externally.
3. `workflow status` reads the stored state.
4. `workflow cancel` updates or terminates the task.

No DBus service, socket server, daemon, or systemd unit is required.

### Future Compatibility

Clients should ignore unknown JSON fields.

Workflow implementations may add additional fields in future API revisions.

This allows the API to evolve without breaking existing plasmoid installations.
